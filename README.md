# 🏛️ [수원시립미술관] 공공기관 웹 접근성 개선 및 리뉴얼 프로젝트

> "추상적인 아이콘의 장벽을 허물고, 명확한 한글 레이블과 상시 노출형 검색 환경을 구현하여 디지털 소외 계층을 배려한 모두를 위한 미술관"

## 1. 프로젝트 소개

- **설명:** 아이콘 중심의 미니멀한 UI로 인해 디지털 숙련도가 낮은 고령층이나 장애인 등 정보 취약계층이 겪던 이용 장벽을 허물고자, 한국형 웹 콘텐츠 접근성 지침(KWCAG)을 준수하여 기획·디자인·개발을 진행한 리뉴얼 프로젝트입니다.
- **문제 정의 (The Core Problems):**
  - **추상적 아이콘의 한계:** 텍스트가 배제된 아이콘이 유발하는 인지적 오해와 해석적 스트레스 완화
  - **심리적 저항선 발생:** 조작 결과의 불확실성으로 인한 실수 및 심리적 위축으로 발생하는 사이트 이탈 방지
  - **검색 접근성 부재:** 메뉴 탐색의 최종 관문인 통합 검색 영역이 아이콘 뒤에 숨어있던 구조적 한계 개선
- **개발 인원:** 개인 프로젝트 (기여도 100%)

## 3. 사용 기술 스택

### Frontend

- **React.js & Vite:** 컴포넌트 기반 아키텍처로 공공기관 웹의 복잡한 모듈을 유지보수하기 용이하게 설계하고 고속 렌더링 성능 구현
- **React Router:** 단일 페이지 애플리케이션(SPA) 환경에서 매끄러운 내비게이션 구현 및 배포 환경에 유동적인 동적 Base URL 설정 적용

### Design & Tools

- **Figma:** KWCAG 표준 체크리스트 및 타겟 인지 가독성을 보장하는 UI/UX 구조 기획 및 프로토타이핑
- **Lighthouse:** 웹 접근성 스코어 정량적 진단 및 시맨틱 마크업 표준 규격 준수 확인 활용

### AI-Assisted Development (Vibe Coding)

- **Google Gemini / Antigravity:** 시맨틱 마크업 구조 검토, 인터랙션 영역의 ARIA 속성 정교화 및 스크린 리더 호환성을 높이는 AI 페어 프로그래밍 기법 도입
- **AI 활용 목표:** 복잡한 공공 정보 위계를 접근성 감리 기준에 맞춰 빠르게 검증하고 마크업 오류 발생율을 사전에 최소화

## 4. 사용자 페르소나 및 과업 흐름 (Task Flow)

기획 단계(PRD)에서 도출한 가상의 취약계층 페르소나의 주요 Pain Point 극복 시나리오입니다.

### 🎯 타겟 페르소나

- **이름:** 박종식 (72세, 은퇴 교사)
- **IT 숙련도:** 하 (유튜브 등의 영상 시청은 가능하나 현대적이고 추상적인 UI 기호나 메타포 이해 불가)
- **Pain Point:** 텍스트가 없는 단순 기호(아이콘) 앞에서 오클릭 시 잘못된 상황이 벌어질까 봐 불안감을 느낌

### 🛠️ 구현한 핵심 시나리오 (Task Flow)

1. **명확한 헤더 레이블 인지:** 상단 메뉴 아이콘 옆에 추가된 한글 텍스트 레이블을 통해 망설임 없이 정확하게 **로그인**이나 **메뉴**로 진입
2. **상시 노출형 검색 수행:** 돋보기를 누르는 허들 없이 바로 화면에 구현되어 있는 **검색 상자**를 통해 원하는 전시 검색 수행
3. **안정적인 시각 피드백 경험:** 호버 및 키보드 포커스 시 확실하게 드러나는 테두리 강조 및 색상 반전을 확인하며 조작에 자신감 획득

## 5. AI 활용 및 개발 워크플로우 (Vibe Coding)

생성형 AI를 적극적으로 활용하여 접근성 표준 준수 효율을 극대화했습니다.

- **초기 구조 설계:** KWCAG 지침을 프롬프트에 반영하여, 시맨틱 태그(header, main, footer, section 등) 중심의 뼈대를 Antigravity를 통해 생성
- **코드 리팩토링:** 비정형적인 div 위주의 코드를 AI 피드백을 통해 웹 접근성에 부합하는 시맨틱 코드로 자동 변환 및 최적화
- **접근성 자동 검사:** 작성된 컴포넌트의 명도 대비(Color Contrast)와 대체 텍스트 적절성을 AI로 1차 검증하여 개발 시간 단축

## 6. 핵심 구현 기능 (Feature Specifications)

기획 명세(PRD)의 우선순위 요건들을 철저하게 코드 레벨에 구현했습니다.

- **헤더 레이블 병기 [UI/P0]:** 주요 유틸리티 아이콘 우측에 14px 이상의 명확한 한글 텍스트 레이블을 병행 배치하여 인지 인프라 확보
- **검색창 실체화 [Search/P0]:** 숨겨진 검색 아이콘 토글 방식을 버리고, 메인 레이아웃에 완전 실체화된 상시 검색 입력 박스 구현
- **시각 피드백 강화 [Access/P1]:** 버튼 및 인터랙티브 요소에 호버(Hover)/포커스(Focus) 시 색상 반전 및 테두리 뚜렷화를 통해 조작 가능성 가독화
- **용어 및 언어 순화 [Content/P1]:** 외래어나 영문 표기를 배제하고 직관적인 '나의 예약', '전시 안내' 등의 우리말 용어 적용
- **웹 표준 접근성 사양 반영 [Dev/P0]:** 전 영역 시맨틱 구조화, 대체 텍스트(`alt`)와 스크린 리더용 `aria-label` 적용으로 웹 접근성 적합성 충족

## 7. 디렉토리 구조

```text
src
├── assets          # 이미지 리소스 및 접근성 지원 디자인 그래픽
├── components      # 접근성이 보장된 공통 및 특화 섹션 컴포넌트 (Header, SearchBar, Footer 등)
├── pages           # 단일 페이지 라우터 타겟 페이지 모음
├── styles          # 고대비 명도 대응 글로벌 스타일
├── App.jsx         # React Router 통합 및 베이스 경로(Basename) 설정
└── main.jsx        # Vite 기반 React 앱 공식 시작 파일
```

## 8. 회고 및 인사이트

**[기술보다 중요한 배려, 웹 접근성]**
웹 접근성을 고려해 페르소나를 설정 페르소나가 겪을 문제들을 임의로 만들어 사이트를 개선하는 과정에서 불필요 한부분을 구분해 제거 하는것이 어려웠습니다.
또한 필요한 부분들로 사이트를 만들어가는 과정에 개인의 창의성이 들어간 디자인이 많이 필요하지 않다는 것을 알게 되었고, 단순하고 쉽게 이용할수 있게 만드는 것이 개발자의 역량임을 알게 되었습니다.

**[AI와의 협업, Vibe Coding의 가능성]**
AI 도구를 단순한 코드 자동 생성이 아닌, 표준 지침 준수 여부를 검증하는 '접근성 감리 파트너'로 활용했습니다. 이 과정에서 프롬프트를 통해 표준 가이드라인을 학습시키고 코드에 적용하는 능력이 실무 효율을 얼마나 높일 수 있는지 체감했습니다.
